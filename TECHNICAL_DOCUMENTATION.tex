\begin{abstract}
This project is a small replication system built with Spring Boot and RabbitMQ. It uses one writer, three replicas, and two reader modes to demonstrate how a distributed system can either favor fast availability or a more reconciled view of the data. The explanation below follows the real code paths in the project and describes what happens in each scenario in natural language.
\end{abstract}

	ableofcontents

\section{Overall Architecture}

The application is packaged as one Spring Boot jar, but the active Spring profile decides which role the process plays. The same code base can run as a writer, as one of the three replicas, or as one of the two reader clients. That profile-driven design keeps the system small while still letting each process behave like a separate node in a distributed system.

The roles are simple:
\begin{itemize}[leftmargin=1.4em]
  \item \textbf{Writer}: publishes new lines into RabbitMQ.
  \item \textbf{Replica1, Replica2, Replica3}: consume write events and persist them locally.
  \item \textbf{ClientReader}: requests the last line and accepts the first available answer.
  \item \textbf{ClientReaderV2}: requests all lines and reconstructs a majority view.
\end{itemize}

The persistent state is not stored in a database. Each replica writes to a plain text file under \texttt{replica\_data/}. That is intentional: it makes divergence visible. When a replica misses a message, its file is immediately different from the others.

\section{How RabbitMQ Is Used}

RabbitMQ is the transport layer that connects producers and consumers. The project uses two fanout exchanges: one for writes and one for read requests. A fanout exchange sends every published message to every queue bound to it, which is exactly what the replication demo needs.

The writer acts as a producer. The replicas act as consumers for writes and also as consumers for read requests. The reader clients are producers of read requests and temporary consumers of the replies. That means the architecture is not a simple one-way pipeline; it is a full request/reply flow built on top of the message broker.

The configuration is straightforward:

\begin{lstlisting}
@Bean
public FanoutExchange writeExchange() {
    return new FanoutExchange(WRITE_EXCHANGE, true, false);
}

@Bean
public FanoutExchange readRequestExchange() {
    return new FanoutExchange(READ_REQUEST_EXCHANGE, true, false);
}
\end{lstlisting}

Each replica profile declares its own write queue and read queue. Those queues are transient and auto-delete, which means they are recreated when the replica starts and disappear when the replica goes away. That choice is important because the project is supposed to show inconsistency visibly rather than silently replay missed messages later.

\section{How a Write Travels Through the System}

When the writer starts, it reads the command-line argument or accepts interactive input and sends the text to the write exchange. The payload is a plain line of text, so the writer does not interpret the content. It only forwards it.

\begin{lstlisting}
private void sendLine(String line) {
    rabbitTemplate.convertAndSend(RabbitMQConfig.WRITE_EXCHANGE, "", line);
    System.out.println("Sent: " + line);
}
\end{lstlisting}

Because the exchange is fanout, RabbitMQ duplicates that same message to all bound replica queues. Each replica then receives the same event independently. There is no leader election and no consensus step in the write path. The system simply relies on the broker to broadcast the message to every consumer queue, and each replica appends the received line to its own local file.

That is why, when all replicas are running, the same line appears in all three files. The data is not copied by one replica to the others. Instead, the broker delivers the message to all consumers, and each consumer persists the line locally.

\section{Replica Behavior}

Each replica starts by computing its own file path and creating the target directory if necessary.

\begin{lstlisting}
@PostConstruct
public void init() {
    if (replicaId != null && !replicaId.isEmpty()) {
        dataFilePath = dataDir + "/data.txt";
        FileUtils.ensureDirectory(dataDir);
        logger.info("Replica {} initialized with data directory: {}", replicaId, dataDir);
    }
}
\end{lstlisting}

This is the reason the three replicas stay independent. Replica1 writes to its own directory, replica2 to another, and replica3 to another. The data files are separate, so a missing message on one node can be inspected directly in the filesystem.

The write listener is equally direct:

\begin{lstlisting}
@RabbitListener(queues = "${replica.write-queue}")
public void handleWrite(String message) {
    logger.info("[{}] Received write message: {}", replicaId, message);
    FileUtils.appendLine(dataFilePath, message);
    logger.info("[{}] Data persisted to {}", replicaId, dataFilePath);
}
\end{lstlisting}

When a replica receives a write event, it simply appends the line and returns. The important point is that the replica does not ask the others whether the write is valid. It does not compare versions. It does not repair itself. It just stores what it received.

Read requests are received on a separate queue. The same replica can answer two different request types: one for the last line and one for the full file.

\begin{lstlisting}
@RabbitListener(queues = "${replica.read-queue}")
public void handleReadRequest(String message) {
    String[] parts = message.split(":", 3);
    String requestType = parts[0];
    String correlationId = parts[1];
    String replyQueueName = parts[2];

    if ("READ_LAST".equals(requestType)) {
        handleReadLast(correlationId, replyQueueName);
    } else if ("READ_ALL".equals(requestType)) {
        handleReadAll(correlationId, replyQueueName);
    }
}
\end{lstlisting}

The correlation identifier is critical because the replies are asynchronous. Without it, a late reply from an older request could be mistaken for the result of a newer one.

\section{ClientReader: Fast Read With First Reply Wins}

	exttt{ClientReader} models an availability-first read. It asks every replica for its last line, but it only needs one answer to finish. This is the simplest possible read strategy in the project and it shows the trade-off clearly: the client gets a quick answer, but it does not attempt any reconciliation.

The client creates a temporary reply queue, starts a listener on that queue, and then sends the request through the read-request exchange.

\begin{lstlisting}
String correlationId = UUID.randomUUID().toString();
String replyQueueName = "reply_" + correlationId;

Queue replyQueue = new Queue(replyQueueName, false, false, true);
rabbitAdmin.declareQueue(replyQueue);

String request = "READ_LAST:" + correlationId + ":" + replyQueueName;
rabbitTemplate.convertAndSend(RabbitMQConfig.READ_REQUEST_EXCHANGE, "", request);
\end{lstlisting}

Each replica that receives the read request looks at its own file and sends back the last line. The first valid response that arrives ends the wait. In practice this means the reader can answer quickly even when the system is under load or one replica is slow.

That behavior is useful because it demonstrates why availability is attractive. The client does not wait for every replica to respond, so it is less sensitive to slow nodes. The trade-off is that it may return a line from a replica that is slightly behind the others.

\section{ClientReaderV2: Full Read and Majority Voting}

	exttt{ClientReaderV2} uses a stronger reconciliation strategy. Instead of asking for only the last line, it asks every replica to send all lines it currently has. The client then collects those replies and reconstructs a merged view of the file.

The reply format contains the replica identifier, the line content, and a final end marker. That lets the client group messages by replica and know when a replica has finished sending its data.

\begin{lstlisting}
Map<String, List<String>> replicaLines = new LinkedHashMap<>();
Map<String, Boolean> replicaEnded = new HashMap<>();
\end{lstlisting}

In natural language, the flow is this: the client broadcasts a read-all request, every replica streams its local file line by line, and the client waits until all replicas report that they are done or until the timeout expires.

The important change in the current version is that majority voting works by line position, not by a numeric prefix. That means the system can reconcile free-form text lines such as \texttt{arbitrary free-form line}. For each position in the file, the client counts the values returned by the replicas and keeps the one that appears in the majority of replicas.

For a three-replica setup, the threshold is two matching replicas. If replica1 and replica3 agree on a line at position 4, that value is kept even if replica2 differs or is missing the line. If only one replica has a value at that position, the line is discarded because it does not have enough support.

This is what makes the readerv2 mode more consistent than the simple reader. It does more work, waits for more data, and then deliberately filters the result using quorum-style reasoning.

\section{Scenario Explanations}

\subsection{Scenario 1: Basic Replication When All Replicas Are Up}

In the healthy case, all three replicas are running before the writer sends anything. When a line is published to the fanout exchange, RabbitMQ copies that message to all three write queues. Each replica consumes the same message and appends it to its local file. Because none of the replicas are down, the three files evolve in the same way.

The result is that each replica shows the same sequence of lines. This scenario proves the normal case: the broker broadcasts the write, the consumers all receive it, and local persistence stays aligned because every node was present to hear the event.

\subsection{Scenario 2: READ\_LAST as an Availability-First Read}

In this scenario, the reader does not try to reconstruct the whole dataset. It sends a read-last request to the broadcast read exchange, and every replica is allowed to answer. The client stops at the first response it receives. The main reason for this design is speed: the reader can complete as soon as one replica is reachable and responsive.

This is why the system can remain available even if one replica is slower or temporarily out of date. The answer might not be globally reconciled, but it is fast and practical. That behavior reflects a common distributed-systems choice: when the user only needs a quick answer, it can be better to return the first valid result instead of waiting for every node.

\subsection{Scenario 3: Replica Failure and Divergence}

When replica2 is stopped, it stops consuming write messages. Replica1 and replica3 continue to receive every published line because they are still connected to RabbitMQ. Replica2, however, is offline, so it misses the new data. After it is started again, it does not magically recover the lines that were published while it was absent.

That is the key point of the scenario: the files now disagree. Replica1 and replica3 contain the newer lines, while replica2 remains behind. The divergence is visible because the project uses local text files instead of an automatic repair mechanism. The behavior is not a bug in the messaging path; it is the expected result of a replica being unavailable when the write happened.

\subsection{Scenario 4: Majority Voting with READ\_ALL}

The readerv2 client demonstrates the reconciliation strategy. It sends a read-all request, gathers the full file contents from every replica, and then compares the replicas line by line. If at least two replicas agree on the content at a given position, that content is kept in the final view.

This scenario explains why the majority-voting client is more consistent than the simple reader. It does not trust the first answer it sees. Instead, it waits for all available replicas and then builds a merged output from the overlapping majority. In a three-node setup, this gives a simple and intuitive notion of truth: the value that appears in at least two replicas is accepted.

\subsection{Scenario 5: Manual Divergence and Reconciliation}

In the final scenario, one replica file is edited by hand to introduce a conflicting line. This simulates a node that drifted from the cluster state. When readerv2 runs afterward, it does not blindly accept that outlier. It compares the values returned by all replicas at the same line position and keeps the value that has majority support.

This scenario is useful because it shows the purpose of majority voting very clearly. A single mismatching replica does not dominate the result. The system prefers the value that the majority of replicas still agree on, which is exactly the kind of reasoning that makes the readerv2 mode more robust than READ_LAST.

\section{What Each Class Does}

\begin{itemize}[leftmargin=1.4em]
  \item \textbf{TpCapReplicationApplication}: starts the Spring Boot application.
  \item \textbf{RabbitMQConfig}: declares the fanout exchanges, replica queues, and bindings.
  \item \textbf{ReplicaService}: consumes writes and read requests, and persists the data locally.
  \item \textbf{ClientWriter}: sends write events to RabbitMQ.
  \item \textbf{ClientReader}: implements the first-reply-wins read path.
  \item \textbf{ClientReaderV2}: implements the full read and majority-voting read path.
  \item \textbf{FileUtils}: handles the filesystem operations used by replicas.
\end{itemize}

\section{Conclusion}

The project is a compact example of how a producer and several consumers can cooperate through RabbitMQ to simulate replication. The writer publishes once, RabbitMQ fans the message out, each replica consumes and persists independently, and the readers demonstrate two different ways to interpret the system state. One reader prefers speed and availability. The other prefers a reconciled answer based on majority agreement. That contrast is the main lesson of the code base, and it is why the scenarios are useful as a CAP-theorem demonstration.
